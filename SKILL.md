# 1. Skill 定位与适用场景

- 将**已经成型的 PRD 文本**（可以是由 `sky-demand-prd-builder-skill` 生成，也可以是任何结构化良好的 PRD）转换为**高保真 HTML+CSS+JS 效果图 / 交互原型**。
- 典型用途：
  - 工具类 / 平台类产品的页面原型验证；
  - 需求评审前快速出一版“接近真 UI”的效果图；
  - 方便产品 / 研发 / 设计在浏览器中直接预览交互路径。
- 只负责：**读 PRD + 生成前端原型文件（HTML+CSS+JS）**。
- 不负责：修改 PRD 文本、后端接口设计、真实数据对接、导出 PDF 或图片（可以由其他工具完成截图或导出）。


# 2. 输入与前置问答（必须先问的平台）

使用本 skill 时，必须先完成一轮简短问答，确定关键参数：

1. **必须询问：产品平台（必填）**
   - 选项说明（文案可以用中文，也可以用英文）：
     - `PC`：桌面端客户端软件（如 Windows / macOS 应用）；
     - `Web`：PC 浏览器中运行的 Web 产品 / 网站；
     - `Mobile`：移动端，包括 iOS / Android 原生应用，以及 H5（手机浏览器中的 Web 页）；
     - `TV`：电视大屏或其他大尺寸显示设备上的应用。
   - 规则：
     - 必须明确只有一个主平台；
     - 仅当用户明确提出“需要多端 / 响应式设计”时，才为多个平台同时生成原型。

2. **可选询问：设计风格 / 主题色（选填）**
   - 若用户愿意提供，可以询问：
     - 品牌色或主色调（例如：蓝色、紫色、黑金等）；
     - 希望风格（例如：简洁、科技、卡片式、极简黑白等）；
     - 是否参考某一设计体系（例如：Material、Ant Design、iOS 风格等）。
   - 若用户不提供任何设计偏好：
     - 由模型根据需求类型选择**一个“默认最美观”的主题**；
     - 后续所有页面必须保持该主题在颜色、字体、组件风格上的一致。

3. **可选询问：移动端偏好参数（当平台=Mobile 时强烈建议问）**
   - iOS / Android / H5（单选为主）；
   - 目标机型宽度或参考尺寸（如 375、390、428）；
   - 是否需要底部 Tab、是否沉浸式状态栏、是否需要底部手势安全区留白。
   - 若用户不提供：
     - 默认按 iOS 风格中性方案生成；
     - 逻辑宽度默认 390，保留安全区内边距。

4. **PRD 文本输入**
   - 来源可以有两种：
     1. 直接传入由 `sky-demand-prd-builder-skill` 生成的 PRD 文本；
     2. 传入任意结构合理的 PRD 文本（Markdown、纯文本等），只要包含页面/模块/流程信息即可。
   - 要求：
     - 若 PRD 中缺少页面或流程说明，要先通过“页面清单草案”与用户确认后再画。


# 3. 职责与边界（只读 PRD，不改 PRD）

- **只读 PRD**：
  - 从 PRD 文本中解析需求背景、用户流程、页面与模块说明、信息架构等；
  - 但**不得**写回或修改 PRD 内容，也不在 PRD 文件中追加任何标记。
- **输出 HTML+CSS+JS 原型**：
  - 输出高保真、可预览的 HTML+CSS+JS 页面集合；
  - 每个页面都有可见的布局、组件与示例文案。
- **不做的事情**：
  - 不负责实现真实业务逻辑（如上传、接口调用、登录等）；
  - 不写后端代码、不设计数据库；
  - 不生成 PDF / 图片导出，只保证 HTML 页面在浏览器中可直接打开和截图；
  - 不强制与任何特定 UI 框架（如 React/Vue）绑定，默认输出原生 HTML+CSS+JS。


# 4. PRD 解析与“页面识别”规则

## 4.1 关键信息提取

从 PRD 中重点提取以下内容：

- 产品范围与目标用户；
- 主要使用场景与关键用户路径（主线流程）；
- 页面 / 模块 / 功能清单；
- 功能之间的关系（导航结构、跳转关系、上下文关系）；
- 重要的状态变化（例如：空状态 / loading / 成功 / 失败等）。

在结构良好的 PRD 中，优先从这些章节名称附近提取信息（名称可以是近义词）：

- “信息架构 / IA / 页面结构 / 导航结构”；
- “主要页面 / 页面列表 / 页面说明 / 模块说明”；
- “用户流程 / 使用流程 / 交互流程 / 任务流”；
- “关键路径 / 典型任务 / 核心流程”。

如果 PRD 没有显式章节，但在正文描述中提到“页面”“界面”“弹框”“卡片”等关键词，也要通过语义识别出相应的页面或 UI 容器。

## 4.3 PRD 质量评估与兜底策略

- 在解析 PRD 时，应先粗略评估其“设计可用性”：
  - 是否包含足够的页面 / 模块 / 流程描述；
  - 是否存在明显冲突（例如同一流程在不同章节描述不一致）；
  - 是否几乎全部是技术实现/接口描述而缺少界面层信息。
- 当 PRD 信息不足以支持“高保真效果图”时，应：
  - 明确告知用户：当前只适合生成“页面结构建议 + 低保真线框级 HTML”，而非完整高保真原型；
  - 或在给出“页面清单草案 + 信息缺口提示”后，请用户补充关键界面信息，再继续生成高保真版本。

## 4.2 页面清单草案与确认

在正式生成 HTML 之前，先构造一份**“页面与路由清单草案”**，包括：

- 页面 ID（英文 slug，如 `select-video`, `subtitle-preview` 等）；
- 页面中文名称（如“选择视频页”“结果预览页”）；
- 页面类型（列表页 / 表单页 / 预览页 / 配置页 / 弹框等）；
- 所在平台（PC / Web / Mobile / TV）；
- 与其他页面的跳转关系（入站来源 / 出站去向）。

然后：

- 将这份清单以可阅读的列表形式展示给用户；
- 明确提示用户可以在此阶段进行“纠偏”（增删页面、调整命名或跳转关系）；
- 若用户确认或明确允许“由模型自行决定”，再进入 HTML+CSS 生成阶段。


# 5. 页面类型与布局映射规则

本 skill 需要内置一套“页面类型 → 布局模板”的映射规则，以保持输出的一致性和可扩展性。示例（可根据需求扩展）：

- **列表页（List Page）**
  - 典型用于：内容列表、任务列表、记录列表等；
  - 布局：顶部导航 / 筛选条 + 列表区域（卡片式或表格式）；
  - 支持分页 / 筛选 / 排序控件。

- **表单 / 配置页（Form / Config Page）**
  - 用于：提交信息、配置参数、创建任务；
  - 布局：分组表单区域 + 操作按钮区域（主操作 / 次要操作）；
  - 需要明显的主按钮（如“开始处理”“保存配置”）。

- **详情 / 预览页（Detail / Preview Page）**
  - 用于：展示单个对象详情、预览结果（如字幕拼接后的图片预览）；
  - 布局：主展示区域 + 侧边信息栏 / 操作栏；
  - 提供“返回”“编辑”“导出”等操作入口（如 PRD 有描述）。

- **仪表盘 / 概览页（Dashboard Page）**
  - 用于：集中展示多个模块的概览数据和入口；
  - 布局：多卡片 / 多面板布局，支持响应式调整。

- **弹框 / 浮层（Modal / Drawer）**
  - 用于：二次确认、填写少量信息、告警提示；
  - 布局：背景遮罩 + 居中或侧边弹出容器，包含标题、正文、按钮组。

- **空状态 / 结果状态页（Empty / Result State）**
  - 用于：无数据、首次使用、操作成功 / 失败后反馈；
  - 布局：插画 / 图标 + 说明文案 + 推荐操作按钮。

页面类型与布局映射时：

- 先根据 PRD 中的“场景 + 功能”推断页面类型；
- 再应用对应模板，并根据平台（PC/Web/Mobile/TV）微调布局（例如：移动端采用单列、TV 采用大字号和栅格布局）。

## 5.1 页面粒度与范围控制

- 为避免超大 PRD 导致一次性生成过多页面，建议：
  - 默认限制在一个合理的上限（例如：不超过 8–12 个主页面 + 若干关键弹框）；
  - 当识别到的潜在页面远超上限时，优先覆盖：
    - 核心业务关键路径相关页面；
    - 高频使用场景页面；
    - 必须进行评审的关键配置/结果页。
- 对于未被覆盖的长尾页面：
  - 可以在“页面清单草案”与“结果摘要”中以 TODO 或“建议由设计/前端补充”的形式列出；
  - 不强行生成粗糙页面，避免噪音过多。


# 6. 视觉与组件体系（统一主题）

## 6.1 统一视觉主题

- 一个 PRD 对应一套视觉主题，主题内容包括：
  - 主色、副色、背景色、警告/成功色；
  - 字体类型与字号体系；
  - 按钮样式（填充、线框、圆角、阴影等）；
  - 卡片、列表、表格等常用容器的样式。
- 若用户未提供主题或品牌要求：
  - 模型要自动选择一套风格统一、对比度良好、可读性强的主题；
  - 并在输出总结中用简短文字说明该主题特点（例如：“科技蓝 + 卡片式布局 + 中性灰背景”）。

## 6.2 视觉规范边界

- 本 skill 输出的是“高保真、可直接预览”的原型视觉：
  - 用于评审、对齐预期、为设计/开发提供参考；
  - 但不等同于完整的品牌视觉规范或正式设计系统交付物。
- 对于已经有品牌手册 / 设计规范的团队：
  - 可以在调用时提供关键约束（主色、字体、圆角风格等），skill 会尽量遵守；
  - 未提供时，则采用通用、美观且中性的默认风格，并明确这一点。

## 6.3 组件与文案示例

- 所有需要显示文字的区域，必须提供语义明确的**示例文案**，而不是“文本占位符”“标题标题”等。
- 常见组件（输入框、按钮、下拉选择、标签、步骤条、进度条等）要在不同页面里保持统一风格。
- 对于列表项、卡片、表格等：
  - 使用真实感较强的示例数据（例如时间、文件名、字幕内容片段等），方便用户一眼看懂页面用途。

## 6.4 组件库 / 设计系统对齐（可选）

- 若用户指定“尽量贴近某个组件库或设计系统”，例如：
  - Material Design；
  - Ant Design；
  - iOS Human Interface Guidelines 等；
- 则在布局与组件命名上尽量靠近对应体系（例如按钮样式、间距、表单布局等），但仍以原生 HTML+CSS+JS 实现；
- 若用户未指定，则使用中性、自研风格组件集合，并在总结中注明“不对齐任何现有组件库”。

## 6.5 美观度硬性规范（必须满足）

- 不能只做到“结构完整”，必须做到“视觉有明确风格方向”：
  - 在输出前先确定一个风格关键词组合（如“轻拟物卡片 + 中性灰 + 品牌主色点缀”）；
  - 所有页面沿用同一风格，不允许页面之间风格断裂。
- 必须建立统一设计 token（至少）：
  - 色彩：主色/辅色/语义色/文本层级色/背景层级色；
  - 圆角：至少 3 档（如 10 / 14 / 20）；
  - 阴影：至少 2 档（卡片、浮层）；
  - 间距：遵循 4/8 递进体系（4、8、12、16、24、32）。
- 必须建立清晰文本层级：
  - 页面标题、区块标题、正文、说明文字至少 4 档；
  - 行高与字号匹配，避免“标题像正文”或“说明字过小不可读”。
- 必须避免“低质原型观感”：
  - 禁止大面积默认浏览器样式按钮/输入框直接裸用；
  - 禁止通篇同一种卡片样式且缺少信息层次；
  - 禁止仅靠一块纯色背景撑全页面（应有层次或轻微质感变化）。

## 6.6 移动端原生感规范（当平台=Mobile 时必须满足）

- 以移动端单手操作为核心：
  - 主要操作按钮优先放在底部固定操作区或拇指易达区域；
  - 关键点击区域最小建议 44×44（逻辑像素）。
- 必须处理安全区与设备差异：
  - 顶部状态栏、底部手势条区域要预留内边距；
  - 底部固定栏需考虑 `safe-area-inset-bottom`。
- 移动端页面结构建议优先：
  - 顶部导航栏 + 主内容滚动区 + 底部操作区；
  - 弹层优先使用 bottom sheet 风格，不默认使用桌面式居中弹窗。
- 交互反馈要“像移动端”：
  - 强调 pressed / active / disabled / loading；
  - hover 仅在 Web 桌面端作为补充，不作为移动端主交互反馈。


# 7. 平台差异化规则

根据前置问答中确定的平台，生成对应端的原型：

- **PC / Web（桌面浏览器端）**
  - 布局：多列栅格、更宽的内容区域，顶部导航或侧边栏导航；
  - 交互：支持 hover、tooltip 等鼠标态；控件密度可以适当提高。

- **Mobile（iOS / Android / H5）**
  - 布局：以单列为主，注意触控区域大小和间距；
  - 加入典型移动端结构（顶部导航栏、底部 Tab 栏等，如 PRD 有相关描述）；
  - 不强调 hover，更多使用 press / active 态；
  - 默认输出中应包含安全区适配和底部固定主操作区（若 PRD 主流程存在关键动作）。

- **TV（电视大屏端）**
  - 布局：大字号、大间距、较少文本；重点突出焦点元素；
  - 交互：强调“焦点态”与“焦点移动路径”，使用高对比度高亮状态标识当前焦点。

若用户明确提出“多端 / 响应式”：

- 需说明每一端的设计差异；
- 可以在同一目录下为不同端生成不同入口页面或使用响应式布局。

## 7.3 响应式与尺寸适配边界

- 默认仅保证典型分辨率下的良好观感：
  - PC/Web：例如 1366×768、1440×900、1920×1080 等常见桌面分辨率；
  - Mobile：常见主流手机宽度（375 / 414 / 428 等逻辑宽度）；
  - TV：常见大屏分辨率（如 1920×1080、3840×2160）。
- 若用户未明确要求“响应式”，则不做复杂自适应布局，仅保证主分辨率下结构合理；
- 当用户明确要求“响应式设计”时：
  - 采用简化的断点方案（例如 mobile / tablet / desktop 三档）；
  - 在 CSS 中清晰标注各断点用途和主要布局差异。

## 7.1 多语言与本地化边界

- 默认仅使用 PRD 文本所在语言（多为中文），不自动生成多语言版本；
- 若 PRD 提到国际化或多语言需求：
  - 可以在原型中预留语言切换控件位置；
  - 但不为每种语言生成完整文案，仅以注释或示例占位体现。

## 7.2 数据与隐私边界

- 所有示例数据均为虚构或适度脱敏数据：
  - 对于手机号、邮箱、订单号等敏感字段，自动采用模糊/假数据；
  - 不从 PRD 或环境中引用用户真实数据。
- 该原型不对接真实后台，不包含真实用户数据读写逻辑，仅作为前端展示与交互示意。


# 8. HTML+CSS+JS 输出规范与目录结构

## 8.1 输出目录

- 所有生成的前端原型文件统一输出到：
  - `prdoutput/ui/`
- 若目录不存在，应在说明中明确“需先创建该目录”，或在脚本实现中自动创建。
- 建议为同一个 PRD 生成时，可以：
  - 在 `prdoutput/ui/` 下创建以 PRD 标题或时间戳命名的子文件夹；
  - 或直接覆盖同名文件，并在结果说明中提醒“本次输出覆盖了旧文件”。

## 8.2 文件命名与结构

- 建议结构（示例）：
  - `prdoutput/ui/index.html`：主入口页面（可为概览或主流程入口）；
  - `prdoutput/ui/<page-id>.html`：各功能页面；
  - `prdoutput/ui/style.css`：全局样式文件；
  - `prdoutput/ui/script.js`：统一的前端脚本文件，用于页面间跳转、高亮导航、基础交互模拟等。

- HTML 规范：
  - 使用语义化标签（`header`, `nav`, `main`, `section`, `article`, `footer` 等）；
  - 在 `<head>` 中设置正确的 `meta charset` 与 `viewport`（按平台差异配置）；
  - 引用统一的 `style.css` 以保证视觉统一。

- CSS 规范：
  - 将主题相关的颜色、字体等抽象为 CSS 变量（例如 `--primary-color`）；
  - 为常用组件定义可复用类（如 `.btn-primary`, `.card`, `.list-item`）。

- JS 使用规范：
  - 仅实现轻量级的前端交互逻辑，如：
    - 页面导航高亮当前页；
    - 控制弹框显示/隐藏；
    - 模拟简单的 loading、步骤切换等；
  - 不实现真实业务逻辑（接口请求、鉴权、长连接等），并在代码中以注释形式说明“此处为模拟交互”。

## 8.3 可访问性与基础无障碍

- 在不显著增加复杂度的前提下，尽量遵守基础无障碍实践：
  - 为关键交互控件添加合适的 `aria-label` 或可见文本说明；
  - 表单控件使用 `<label>` 与输入框关联，方便键盘与读屏使用；
  - 保证文本与背景之间有足够的颜色对比度（避免浅灰文字配浅背景）；
  - 为图标按钮提供文字替代或 `title`/`aria-label` 说明其含义。
- 不追求完全符合某一无障碍标准（如 WCAG 全量），但应避免明显“不可用”或“看不清”的设计。

## 8.4 输出质量补充物（建议强制）

- 除 HTML/CSS/JS 文件外，建议同时输出一个简短说明文件（如 `ui-spec.md`），至少包含：
  - 视觉主题说明（风格关键词、主色、背景策略）；
  - 页面清单与主流程跳转链路；
  - 与 PRD 的映射关系摘要（哪些需求点映射到哪些页面模块）。
- 若是移动端原型，`ui-spec.md` 还应注明：
  - 目标逻辑宽度；
  - 导航模式（顶部栏/底栏/Tab）；
  - 主要操作区在页面中的位置与原因。


# 9. 交互态与页面跳转

- 所有有交互的元素，需在 CSS 中定义状态样式：
  - `:hover`, `:active`, `.disabled`, `.loading` 等；
  - 并在 HTML 结构或注释中标明状态用途（例如：“按钮 disabled 时不可点击，灰色背景”）。

- 页面跳转：
  - 使用标准的 `<a href="xxx.html">` 或按钮包裹链接；
  - 至少覆盖 PRD 中描述的主流程路径（如“选择视频 → 配置参数 → 处理进度 → 结果预览”）。

- 对于弹框 / 浮层等：
  - 可以在同一页面内用额外容器实现，配合 CSS 类控制显示 / 隐藏；
  - 或单独拆成页面，并在注释中说明“实际实现时可使用弹框形式”。

- 复杂交互上限说明：
  - 本 skill 仅模拟“轻交互”，包括页面跳转、步骤切换、loading/disabled 等基础状态；
  - 不实现复杂画布操作、自由拖拽排版、时间线拖动、多窗口联动等高复杂度交互；
  - 对于此类需求，会通过静态状态示例 + 注释方式表达交互意图。

- 微动效与转场建议（提升“高保真感”）：
  - 页面加载时允许轻量入场动效（如卡片淡入上移）；
  - 按钮按下有明确视觉反馈（缩放/阴影/亮度变化其一）；
  - 弹层出现/收起有 180–280ms 的平滑过渡；
  - 禁止花哨且不服务任务的动画堆叠。


# 10. 纠偏与迭代机制

本 skill 必须为用户保留清晰的“纠偏入口”：

- 在首次生成原型后，向用户输出一份**结果摘要**，包括：
  - 已生成的页面列表（ID + 中文名称 + 类型）；
  - 使用的平台与视觉主题概述；
  - 主要交互路径说明。

- 明确提示用户可以纠偏的内容包括但不限于：
  - 增删页面或调整页面层级、命名；
  - 调整主题色、字体风格、组件风格；
  - 调整某个页面的布局（如将双列改为单列、增加侧边栏等）；
  - 修改示例文案，使之更贴近业务。

- 在后续迭代中：
  - 应尽量**仅调整用户指定的页面 / 区域**，保持其他页面和整体视觉风格不变；
  - 在说明中标出本次迭代主要改动点，帮助用户快速核对。

## 10.1 版本追踪与可追溯性

- 在生成的 HTML 文件中（例如 `<head>` 注释或开头注释），建议包含：
  - 对应 PRD 的标题（或摘要标识）；
  - PRD 版本号或生成时间戳；
  - 目标平台信息（PC/Web/Mobile/TV）。
- 当 PRD 版本更新时，本 skill 可以：
  - 先识别 PRD 的变更范围（新增/修改的模块与流程）；
  - 只针对受影响的页面进行局部调整，避免每次都完全推翻之前的原型。

## 10.2 HTML 与 PRD 的一致性审查机制

- 在生成 HTML+CSS+JS 原型后，本 skill 应进行一次“自审查”步骤，检查原型与 PRD 的一致性，并输出检查结果摘要，包括但不限于：
  - **页面覆盖情况**：PRD 中识别出的页面 / 模块，是否都有对应的 HTML 页面或区域；
  - **导航与流程**：核心用户路径（如 PRD 中的主流程步骤）是否都有可达的页面跳转链路；
  - **关键元素与操作**：PRD 明确提出的重要操作（按钮、入口、配置项）是否在页面中有体现；
  - **状态示例**：对需要强调的状态（loading / 空状态 / 成功 / 失败等）是否提供了至少一个示例。
- 若发现以下情况，应在结果中显式标出，并以“TODO / 待确认”等方式提示：
  - 某些 PRD 中出现的页面/模块尚未在原型中体现；
  - 原型中出现了 PRD 中未提及、且无法合理推断的“超出范围页面”；
  - 关键路径中存在“断点”（从某页面无法继续走完主流程）。
- 审查机制的目标是“辅助发现偏差”，不能保证 100% 形式化一致，但能帮助产品/设计快速聚焦需要人工核对的部分。

## 10.3 美观与可用性验收清单（新增）

生成完成后，必须按以下清单做一次自检，并在结果摘要中给出“通过/待改进”：

- 视觉一致性：
  - 是否存在统一 token 与组件风格；
  - 是否存在明显风格断裂页面。
- 移动端可用性（平台=Mobile）：
  - 关键操作是否在易达区域；
  - 触控尺寸与间距是否足够；
  - 安全区是否正确处理。
- 信息层级：
  - 关键 CTA 是否突出；
  - 页面主次信息是否一眼可分辨。
- PRD 对齐度：
  - PRD 的核心流程是否可完整走通；
  - PRD 明确要求的关键配置项/状态是否在原型中体现。

若任一项明显不满足，不应直接结束，而应先进行一次自动修正再输出。

## 10.4 PRD 映射矩阵（新增）

- 在结果摘要中附一个简短“需求点 → 页面/模块”的映射表（3 列即可）：
  - PRD 需求点；
  - 对应页面 ID / 区块；
  - 当前实现状态（已覆盖 / 部分覆盖 / 待补充）。
- 映射表的目的是保证“美观”不脱离“需求”，防止只做视觉包装而遗漏关键功能表达。


# 11. 与 PRD Skill 的衔接示例

- 推荐的上游流程（示例）：
  1. 使用 `sky-demand-evaluator` 评估用户需求真伪与优先级；
  2. 对于真需求，使用 `sky-demand-prd-builder-skill` 生成结构化 PRD 草稿；
  3. 在 PRD 经过人工补充与确认后，将最终 PRD 文本作为输入调用本 skill；
  4. 本 skill 生成对应平台的 HTML+CSS+JS 高保真原型，输出到 `prdoutput/ui/`。

- 同时，本 skill 也支持直接使用用户已存在的 PRD 文档，只要包含足够的页面与流程描述。


# 12. 可扩展性建议

为了便于后续扩展和与其他工具联动，本 skill 在设计上应满足：

- **页面类型可扩展**：
  - 可以在现有模板基础上新增更多页面类型（如多步骤表单、时间线视图等）；
  - 新类型同样遵循“页面类型 → 布局模板”的映射模式。

- **视觉主题可扩展**：
  - 支持选择不同预设主题（如“企业蓝”“暗黑科技”“极简白”等）；
  - 甚至根据品牌主色自动生成一套衍生色板。

- **输出可被其他工具消费**：
  - 生成的 HTML+CSS+JS 结构清晰，便于：
    - 设计工具将其作为参考图导入；
    - 开发直接在此基础上改造为真实前端项目；
    - 其他脚本对其进行截图、导出 PDF 等操作。

- **与自动化流程集成**：
  - 该 skill 的“输入→生成→审查→纠偏”链路适合被 CI/自动化流程调用；
  - 例如：PRD 更新后自动触发原型重生成与一致性审查，再由人工确认是否进入开发阶段。

- **预期管理与使用建议**：
  - 明确强调：本 skill 旨在“缩短从 PRD 到可见界面的距离”，帮助沟通和评审；
  - 不替代专业 UX 研究、可用性测试和最终视觉设计定稿；
  - 当 PRD 含糊或冲突时，会通过注释方式标记假设与风险点，等待产品/设计确认后再迭代。


# 13. 示例：字幕截帧拼图需求的页面映射

以“从有字幕的视频中截取字幕帧 → 去重 → 拼接成一张长图”的工具型功能为例，本 skill 对 PRD 的典型映射如下：

- **1）从 PRD 中识别出的核心页面**
  - 选择视频页：用户首次进入时选择本地视频或拖拽导入，展示基础说明与格式要求；
  - 任务配置 / 进度页：配置是否去除相同字幕帧、截取区域、输出分辨率等，并展示处理进度（队列状态、当前帧进度）；
  - 结果预览页：展示最终拼接好的长图预览，可放大/缩小、滚动查看，并提供导出/保存入口；
  - （可选）异常与帮助入口：用于展示识别失败、字幕区域不清晰等情况下的提示和解决建议。

- **2）可能生成的文件结构（示例）**
  - `prdoutput/ui/index.html`：作为“选择视频页”，包含文件选择区、近期任务入口以及简要说明；
  - `prdoutput/ui/task-config.html`：任务配置与处理进度页，包含参数表单、进度条、任务状态列表；
  - `prdoutput/ui/result-preview.html`：结果预览页，展示长图预览、缩放控制和导出按钮；
  - `prdoutput/ui/style.css`：统一的工具型视觉主题（如科技蓝 + 深色背景 + 卡片式布局）；
  - `prdoutput/ui/script.js`：处理页面间跳转、导航高亮、模拟任务进度的前端脚本。

- **3）页面跳转与流程示意**
  - 用户访问 `index.html`，选择视频 → 点击“开始截取字幕” → 跳转到 `task-config.html`；
  - 在 `task-config.html` 中配置参数并点击“开始处理”，页面进入 loading/处理中状态；
  - 处理完成后展示“查看结果”按钮，点击后跳转到 `result-preview.html`；
  - `result-preview.html` 中支持返回操作，用户可回到配置页重新调整参数。

- **4）一致性审查在本示例中的体现**
  - 针对“字幕截帧拼图”这个具体 PRD，审查会检查：
    - 该 PRD 中是否还提到其他页面（如“任务历史”“系统设置”等）尚未在 HTML 中表现；
    - 示例主流程“选择视频 → 配置/处理 → 查看结果”在页面跳转链路中是否完整可达；
    - 示例 PRD 强调的关键元素（如“去除相同字幕帧”的开关、“字幕截取区域”的配置项）是否在页面中有对应控件。
  - 对于其他任意 PRD，使用相同思路：检查“该 PRD 自己定义的主流程”和“该 PRD 自己强调的关键元素”是否在原型中被覆盖，而不是固定套用本例中的具体流程或字段。
